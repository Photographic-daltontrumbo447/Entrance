# 2.1 Erlang OTP Supervisor Model Design

> Status: mounted hot detail (transitional)

## Purpose

- define the supervision model as one of the first top design documents
- define how retry, reporting, and escalation are represented structurally
- define the mounting point for runtime supervision strategy and runtime-enforcement follow-up

## Mounted Root

- [machine.md](./machine.md)

## Confirmed Oracle Points

- This document preserves the supervision-binding cut beneath [machine.md](./machine.md): typed runtime signals, supervisor actions, retry eligibility, and hot projection.
- The supervision slogan remains `max_retry + report + no_silent_failure`.
- Correctness depends first on strict `state / route / writer / receipt / capability` boundaries; retry budget is a fuse rather than the main anti-drift mechanism.
- Every failure must remain visible; recovery must not erase incident visibility.
- Supervision remains runtime-owned control plus hot projection rather than a fourth canonical state family at v0.
- The v0 typed runtime signal families stay `EXECUTION_FAILURE_SIGNAL / ADMISSION_REJECTION_SIGNAL / VERDICT_RETURN_SIGNAL / INTEGRITY_SIGNAL`.
- The minimal v0 supervisor action set stays `RESTART_CHILD / REPLACE_CHILD / ROUTE_RETURN / BLOCK_LINEAGE / QUARANTINE_LINEAGE / ESCALATE_UP / SURFACE_INCIDENT`.
- Only `EXECUTION_FAILURE_SIGNAL` may consume automatic retry budget at v0; admission rejection, verdict return, and integrity signals remain retry-ineligible by default.
- Ordinary accepted or repairable verdict returns should flow through runtime return routing rather than being misclassified as restartable supervision failures.
- The restart or replace target should stay at execution-child scope; dispatch pipelines and session bundles remain routing or admin envelopes rather than normal retry units.
- v0 supervision enforcement still follows one slice order: classify typed signal, append incident visibility, compute supervisor action, apply retry budget only on the execution-restart path, update overlays when required, then route return or escalation.
- The minimum hot supervision surface remains `current_supervision_state / retry_count / last_failure_signal_family / last_failure_code / last_supervisor_action / escalation_pending`.
- Retry exhaustion should project `Blocked` at v0; `Failed` remains a UI alias rather than a separate projected supervision state.

## Current Boundary Reading

- retry policy and visibility policy should stay coupled rather than being designed separately
- replace should create a newly compiled instance rather than resume a tainted mid-state
- supervision should bind to typed runtime signals rather than ad hoc task strings
- retry budget applies to execution failure recovery, not to ordinary semantic rejection loops
